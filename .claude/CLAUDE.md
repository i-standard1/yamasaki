## Claude Code 設定

### 必須設定
- `.claude/skills/` に開発ワークフローのスキルを同梱済み（自動で判定・実行される）
- `.claude/rules/` に条件付き自動ロードルールを配置済み。ファイル操作時に該当ルールが自動ロードされる：
  - `spec-management.md`: Spec・設計書操作時。ID体系・spec-map・品質ゲート・依存グラフ
  - `implementation.md`: ソースコード操作時。コード探索・デザイン方針・自動化・コンポーネント管理
  - `git-workflow.md`: Git操作時。ブランチ戦略・コミット規約・状態定義
  - `session-ops.md`: タスク・ログ操作時。セッション復帰・ワークフロー設計・ヘルスチェック・エラーリカバリー
  - `i18n.md`: 国際化ファイル操作時
  - `pm-orchestration.md`: docs/ 操作時。PM主導スキルの共通パターン・統一基盤チェックリスト
  - `supabase.md`: supabase/ 操作時。Supabase Auth + RLS 実装ルール
  - `testing.md`: tests/, *.test.* 操作時。テスト戦略 + E2Eフレームワーク検出
  - `release-and-branching.md`: .github/, ci-templates/ 操作時。リリース管理・ブランチ戦略・競合回避
  - `doc-accuracy.md`: docs/ 操作時。ドキュメント正確性・サイトスケーラビリティ
  - `async-teams.md`: domains/, logs/, tasks/ 操作時。非同期チーム・ハンドオフ
  - `design-handoff.md`: docs/designs/ 操作時。デザインファイル管理ルール
  - `security.md`: 機密ファイル, credentials, ssh, deploy 操作時。セキュリティポリシー

### ルール優先順位（複数ルールが同時発火した場合）
1. **security.md** — セキュリティは常に最優先。他のルールと矛盾する場合はセキュリティが勝つ
2. **spec-management.md** — Spec体系・品質ゲートはプロセスの骨格
3. **implementation.md / testing.md** — コード・テストの実装ルール
4. **その他のルール** — 状況に応じて適用

### セキュリティ設定
詳細は `.claude/rules/security.md` を参照（機密ファイル操作時に自動ロード）。
概要: サンドボックス有効、deny ルールで破壊的コマンドをブロック、PreToolUse フックで本番接続を検出。

### モデル設定
Claude Codeの `/model` コマンドで以下を推奨設定にすること：
- **プランモード（設計・判断）**: Opus
- **実行モード（コード生成）**: Sonnet

### テンプレート設計原則
`docs/templates/` 改修時は厳守：
- **1テンプレート = 1関心事 = 1サブエージェント担当範囲**
- **フェーズ境界を守る** — phase1（WHAT）と phase2（HOW）を混ぜない

## AI主動開発ルール

### 基本原則
- **AIが主導、人間は判断**
- **設計書ファースト** — コード変更前に設計書を更新。実装後は設計書⇔コードの一致を検証し、不一致は同コミットで修正
- **次のステップを提案** — 作業完了後は「次は○○しますか？」と提示

### コア原則
- **シンプル第一**：変更・影響範囲を最小に
- **手を抜かない**：根本原因を特定し、シニアエンジニア水準を維持
- **影響を最小化**：必要箇所のみ変更。バグを持ち込まない

### 新規プロジェクト（0→1）の開発順序
全機能の要件定義 → 基本設計 → 人間の確認 → 実装。途中で「実装して」と言われても、要件定義・基本設計の確定を先に提案すること。

### マルチエージェント構成
実装を伴うリクエストは orchestrate（SuperPM→ドメインPM→PG の3層）が自動選択される。ドキュメント生成のみ（spec-feature / draft-spec 等）では動かない。`domains/` 未存在時は自動初期化。
