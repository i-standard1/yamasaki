# フェーズ 3: Spec 執筆（メインが一元管理）

サブエージェントの分析結果をもとに、**メインエージェントが Spec を執筆する**。
サブエージェントに Spec 執筆を委任してはならない。

### 各バッチ開始時に必ず読むファイル（統一性の担保）

コンテキスト圧縮で前のバッチの内容が消えても、以下のファイルを読めば統一基準を維持できる：

1. `docs/requirements/glossary.md` - 用語辞書（表記ゆれ防止）
2. `logs/context/specall_reqid_plan.md` - REQ-ID カテゴリ一覧（ID 重複防止）
3. [Spec 執筆標準](../../_shared/spec-writing-standard.md) - セクション構成・記述ルール
4. `docs/requirements/overview.md` - 最新の機能一覧・ステータス（進捗把握）

**これにより、PM の記憶ではなくファイルが統一性の源泉になる。**

### 3.1 バッチ処理

コンテキスト圧縮を避けるため、バッチに分けて処理する。
**バッチサイズは固定しない。** 以下の基準で動的に判断する：

- 分析レポートのボリュームを見て、コンテキストに余裕がある限り詰める
- 大規模機能（モデル行数 500行超、コントローラアクション 10超）→ 1〜2 機能/バッチ
- 中規模機能（モデル行数 100〜500行）→ 3〜5 機能/バッチ
- 小規模機能（モデル行数 100行未満、コントローラアクション 3以下）→ 5〜10 機能/バッチ
- **トークン残量が 7万以下なら 1機能ずつ処理する**（統一基盤ファイル常駐分のバッファを考慮）

各機能について：
1. サブエージェントのレポートから該当機能の分析結果を読む
2. **不足があれば自分でコードを追加読みする**（レポートだけで判断しない）
3. `docs/requirements/features/[機能名].md` を生成する

### 3.2 Spec フォーマット

spec-feature のフェーズ3と同じルールを適用する：
- REQ-ID 採番（REQ-[機能]-[連番]）
- 受入条件（具体的な数値を明記、チェックリスト形式）
- ビジネスロジック（Mermaid 図で視覚化）
- 全記述にコード根拠（ファイル名:行番号）を付与
- 非機能要件
- **要件定義書にはWHAT（何を作るか）だけを記載する。** 画面遷移図・APIエンドポイント・データモデルは基本設計書に記載済みのため含めない

### 3.3 付随更新

各バッチで以下も更新する：
- `docs/requirements/overview.md` の REQ-ID・ステータスを更新
  - Spec 執筆完了 → ステータスを「完了」に変更
  - 次バッチ以降で着手予定 → 「未着手」のまま
- `spec-map.yml` にエントリ追加
- `mkdocs.yml` の nav に機能ページを追加（**フェーズ1.4で決定した階層構成に従う**。ドメイン別カテゴリの中に配置する）
- `docs/requirements/glossary.md` に新しい用語があれば追記

### 3.4 バッチごとのコミット

**各バッチの Spec 執筆 + 付随更新が終わったら必ずコミットする。**
セッション中断時に未コミットの成果物を失わないため。

```
feat: spec-all バッチN完了（REQ-XXX, REQ-YYY, REQ-ZZZ）
```

### 3.5 バッチ完了時レビュー

**各バッチ完了後、統一性レビューエージェントを起動する。**

[Spec 統一性レビュー](../../_shared/spec-consistency-review.md) に従い、
サブエージェントに全既存 Spec の横断チェックを実行させる。

レビュー結果への対応：
- **自動修正可能** → 即座に修正して次のバッチへ
- **要確認** → ユーザーに提示して判断を仰ぐ（次のバッチは判断待ちせず並行で進めてよい）
