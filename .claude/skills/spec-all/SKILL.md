# 全機能一括 Spec 化（PM オーケストレーション）

PM オーケストレーション共通パターン（`.claude/rules/pm-orchestration.md` で自動ロード）に従い、
メインエージェントが PM として Spec を一元執筆する。

## 前提条件

- `docs/requirements/overview.md` に機能一覧が存在すること（init-spec 済み）
- コードリポのパスが `CLAUDE.md` の「プロジェクト構成」に記載されていること

---

## フェーズ 0: 統一基盤の確立

Spec 執筆を始める前に、統一ルールをファイルに永続化する。
**PM のコンテキストが圧縮されても、各バッチがこのファイルを読めば統一性を維持できる。**

### 0.1 用語辞書の作成

1. コードリポの主要モデル名・コントローラ名を一覧化する
2. それぞれに対応する日本語用語を決定する
3. `docs/requirements/glossary.md` に「用語 / 定義 / コード上の表現 / 使わない表記」のテーブル形式で書き出す
4. ユーザーに確認する：「この用語統一で進めますか？」

### 0.2 統一基盤の確認

統一基盤確認チェックリスト（`.claude/rules/pm-orchestration.md` で自動ロード）を実施する。
特に [Spec 執筆標準](../_shared/spec-writing-standard.md) のセクション構成・記述ルール・REQ-ID 規則を把握する。

### 0.3 REQ-ID カテゴリの事前採番

overview.md の全機能に対し、REQ-ID のカテゴリ部分を事前に決定する：

```
ユーザー認証・登録 → REQ-AUTH
ユーザープロフィール → REQ-PROFILE
案件管理 → REQ-PROJECT
...
```

これを `logs/context/specall_reqid_plan.md` に記録する。
各バッチはこのファイルを参照して REQ-ID を採番する。

## フェーズ 1: ドメイン分割計画

### 1.1 現状把握
1. `docs/requirements/overview.md` の機能一覧を読む
2. コードリポのディレクトリ構造を確認（CLAUDE.md の技術スタックと [技術スタック対応ガイド](../_shared/tech-stack-guide.md) のバックエンド別パス対応表から、モデル層・コントローラ層のパスを特定する）
3. ルーティング定義（例: Rails=`config/routes.rb`, Django=`*/urls.py`, Next.js=`app/`ディレクトリ構造）を読み、機能とファイルの対応を把握する

### 1.2 ドメイン分割
機能一覧をドメイン（3〜6個）に分割する。分割基準：
- **コントローラの名前空間**に合わせる（`admins/users/` → ユーザードメイン等）
- 1ドメインあたり 3〜6 機能が目安
- 関連モデルが共通する機能は同じドメインにまとめる

**大規模ドメインの分割ルール:**
- 対象ファイルの合計行数が **3,000行超** または **機能数が6超** のドメインは、サブエージェントのコンテキスト枯渇を防ぐため **最初から2つのサブドメインに分割** する
- 分割の粒度: 機能の依存関係が薄い境界で分ける（例: 「ユーザー認証・登録」と「ユーザープロフィール・検索」）
- 分割後の各サブドメインは独立したサブエージェントとして起動する
- 計画記録時に分割理由と分割境界を明記する

### 1.3 ファイル割り当て
各ドメインに対し、分析対象ファイルを**明示的に列挙**する：

**バックエンド（[技術スタック対応ガイド](../_shared/tech-stack-guide.md) のバックエンド別パス対応表を参照）:**
- モデル層（例: Rails=`app/models/xxx.rb`, Django=`*/models.py`, Go=`*/model/*.go`）
- コントローラ層（例: Rails=`app/controllers/xxx/`, Django=`*/views.py`, Express=`*/controllers/`）
- 関連するメーラー、ジョブ、アップローダー
- ルーティング定義の該当セクション

**フロントエンド（[技術スタック対応ガイド](../_shared/tech-stack-guide.md) のフロントエンド別パス対応表を参照）:**
- ページコンポーネント（例: Next.js App Router=`app/**/page.tsx`, Nuxt=`pages/**/*.vue`）
- 関連する UI コンポーネント（`components/`）
- 状態管理（Redux slice、Pinia store 等）
- API 通信ユーティリティ（`utils/`, `api/`, `services/` 等）

**共通（全ドメインにスキーマ定義を割り当て）:**
- スキーマ定義（例: Rails=`db/schema.rb`, Django/SQLAlchemy=models.py, Prisma=`prisma/schema.prisma`）を対象ファイルに含める
- 各エージェントは自ドメインのテーブルのみ抽出する

**ドメイン横断モデルの扱い:**
- 複数ドメインにまたがるモデル（例: contract.rb がユーザーにも案件にも関連）は **主たるドメインに割り当て、他ドメインには「共有モデル」として参照指示を出す**
- `logs/context/specall_plan_[timestamp].md` の「共有モデル一覧」セクションに記録し、各分析エージェントに通知する
- 例: `contract.rb → 案件・契約ドメインが主担当。ユーザードメインのエージェントは contract.rb のうちユーザー関連の association/scope のみ抽出`

### 1.4 ナビゲーション階層の決定

ドメイン分割をそのまま `mkdocs.yml` のナビゲーション階層に使う。
**全機能をフラットに並べない。** ドメインをカテゴリとして階層化する（例: `要件定義 > ユーザー管理 > 認証・登録`）。
カテゴリ名は日本語で、ユーザーが直感的に探せる名前にする。計画記録時にナビゲーション構成案も含める。

### 1.5 計画記録
`logs/context/specall_plan_[timestamp].md` に以下を記録：
- ドメイン一覧と各ドメインの担当機能
- 各ドメインの分析対象ファイル一覧
- **共有モデル一覧**（どのモデルがどのドメインの主担当か、他ドメインは何を参照するか）
- **ナビゲーション階層構成案**（ドメイン→機能の対応）
- バッチ実行順序（依存関係がある場合）

### 1.6 ユーザー確認
ドメイン分割計画（ドメイン名・機能数・主要ファイル）をユーザーに提示し、承認を得る。

## フェーズ 3: Spec 執筆（メインが一元管理）

サブエージェントの分析結果をもとに、**メインエージェントが Spec を執筆する**。
サブエージェントに Spec 執筆を委任してはならない。

### 各バッチ開始時に必ず読むファイル（統一性の担保）

コンテキスト圧縮で前のバッチの内容が消えても、以下のファイルを読めば統一基準を維持できる：

1. `docs/requirements/glossary.md` - 用語辞書（表記ゆれ防止）
2. `logs/context/specall_reqid_plan.md` - REQ-ID カテゴリ一覧（ID 重複防止）
3. [Spec 執筆標準](../_shared/spec-writing-standard.md) - セクション構成・記述ルール
4. `docs/requirements/overview.md` - 最新の機能一覧・ステータス（進捗把握）

**これにより、PM の記憶ではなくファイルが統一性の源泉になる。**

### 3.1 バッチ処理

コンテキスト圧縮を避けるため、バッチに分けて処理する。
**バッチサイズは固定しない。** 以下の基準で動的に判断する：

- 分析レポートのボリュームを見て、コンテキストに余裕がある限り詰める
- 大規模機能（モデル行数 500行超、コントローラアクション 10超）→ 1〜2 機能/バッチ
- 中規模機能（モデル行数 100〜500行）→ 3〜5 機能/バッチ
- 小規模機能（モデル行数 100行未満、コントローラアクション 3以下）→ 5〜10 機能/バッチ
- **トークン残量が 7万以下なら 1機能ずつ処理する**（統一基盤ファイル常駐分のバッファを考慮）

各機能について：
1. サブエージェントのレポートから該当機能の分析結果を読む
2. **不足があれば自分でコードを追加読みする**（レポートだけで判断しない）
3. `docs/requirements/features/[機能名].md` を生成する

### 3.2 Spec フォーマット

spec-feature のフェーズ3と同じルールを適用する：
- REQ-ID 採番（REQ-[機能]-[連番]）
- 受入条件（具体的な数値を明記、チェックリスト形式）
- ビジネスロジック（Mermaid 図で視覚化）
- 全記述にコード根拠（ファイル名:行番号）を付与
- 非機能要件
- **要件定義書にはWHAT（何を作るか）だけを記載する。** 画面遷移図・APIエンドポイント・データモデルは基本設計書に記載済みのため含めない

### 3.3 付随更新

各バッチで以下も更新する：
- `docs/requirements/overview.md` の REQ-ID・ステータスを更新
  - Spec 執筆完了 → ステータスを「完了」に変更
  - 次バッチ以降で着手予定 → 「未着手」のまま
- `spec-map.yml` にエントリ追加
- `mkdocs.yml` の nav に機能ページを追加（**フェーズ1.4で決定した階層構成に従う**。ドメイン別カテゴリの中に配置する）
- `docs/requirements/glossary.md` に新しい用語があれば追記

### 3.4 バッチごとのコミット

**各バッチの Spec 執筆 + 付随更新が終わったら必ずコミットする。**
セッション中断時に未コミットの成果物を失わないため。

```
feat: spec-all バッチN完了（REQ-XXX, REQ-YYY, REQ-ZZZ）
```

### 3.5 バッチ完了時レビュー

**各バッチ完了後、統一性レビューエージェントを起動する。**

[Spec 統一性レビュー](../_shared/spec-consistency-review.md) に従い、
サブエージェントに全既存 Spec の横断チェックを実行させる。

レビュー結果への対応：
- **自動修正可能** → 即座に修正して次のバッチへ
- **要確認** → ユーザーに提示して判断を仰ぐ（次のバッチは判断待ちせず並行で進めてよい）

[spec-all 実行詳細](./execution.md) の「フェーズ 4: 設計書統合更新」に従い、サブエージェントで情報を再収集してから基本設計書を1ファイルずつ更新・コミットする。

---

## フェーズ 4.5: 共通コンポーネント設計書の生成

全ドメインのコード分析結果を横断的に集約し、`docs/design/shared-components.md` を生成する。

1. フェーズ2のサブエージェント分析結果から、複数ドメインで共有されているコンポーネントを特定する:
   - UIコンポーネント（`components/` 配下で複数ページから参照されるもの）
   - ユーティリティ関数（`lib/`, `utils/` 配下で複数機能から呼ばれるもの）
   - カスタムフック、ミドルウェア、認証ガード等
2. テンプレート（`docs/templates/phase2/shared-components.md`）のセクション構成に従い、共通コンポーネント設計書を生成する
3. 各コンポーネントにコード根拠（ファイル名:行番号）を付与する
4. フェーズ3で生成した各 feature-design.md のセクション3「共通コンポーネント」に、利用する共通コンポーネントを記載する
5. → コミット: `docs: 共通コンポーネント設計書を生成 (shared-components.md)`

---

## フェーズ 5: 依存グラフ生成

全Spec完成後、`docs/design/dependency-graph.md` を生成または更新する：

1. `spec-map.yml` の全エントリから `depends_on` を読み取り、依存関係を集約する
2. Mermaid graph TD で依存グラフを描画する
3. 依存マトリクス（テーブル形式）を生成する
4. 循環依存チェックを実行し、検出された場合は警告を提示する
5. → コミット: `docs: Spec依存グラフを生成 (dependency-graph.md)`

---

## フェーズ 6: 最終レビュー

全ドキュメント完成後、2種類のレビューを実行する：

### 6.1 統一性レビュー
1. [Spec 統一性レビュー](../_shared/spec-consistency-review.md) をフルチェックモードで起動
2. レビュー結果を全て反映
3. `docs/requirements/glossary.md` に未登録の用語があれば追記

### 6.2 仕様カバレッジレビュー（コード分析モードのみ）
1. [仕様カバレッジレビュー](../_shared/spec-coverage-review.md) に従い、コードと要件定義書を突合
2. 漏れがあれば該当Specに追記してコミット

### 6.3 ビルド確認
`mkdocs build` でビルドエラーがないことを確認

---

## フェーズ 7: 完了

1. コミット（ドメイン分割計画、全 Spec、設計書更新を含む）
2. 成果サマリーを提示（Spec生成数、REQ-ID件数、設計書更新一覧、レビュー修正件数）
3. 次のステップを提案（「実装に入りますか？」「テストを作りますか？」等）

---

## セッション中断時の再開

セッションが途中で切れた場合、新しいセッションで「spec-all を再開して」と言えば、
以下のファイルから進捗を復元できる：

1. `logs/context/specall_plan_*.md` → ドメイン分割計画・バッチ順序
2. `logs/context/specall_reqid_plan.md` → REQ-ID カテゴリ一覧
3. `docs/requirements/overview.md` → ステータスが「未着手」以外の機能 = 完了済み
4. `docs/requirements/features/` → 既に生成された Spec ファイル
5. `docs/requirements/glossary.md` → 用語辞書

**再開手順:**
1. 上記ファイルを読み、完了済みバッチと未完了バッチを特定する
2. 未完了の最初のバッチからフェーズ3を再開する
3. 分析エージェントの結果が logs/context/ に残っていれば再利用する。なければフェーズ2から再実行する

## トークン節約ルール

PM オーケストレーション共通パターン（自動ロード済み） の「トークン節約ルール」に従う。
追加ルール：
- バッチサイズは機能規模に応じて動的に決定し、コンテキスト圧縮を防ぐ

## ルール

- PM オーケストレーション共通パターン（自動ロード済み） の全ルールに従う
- **実装に書いていないことは書かない。** spec-feature と同じ正確性ルールを適用する
- **統一性レビューを省略しない。** バッチごと + 最終の両方で必ず実行する
- 既存コードは一切変更しない

**コード分析時の注意:** .claude/rules/doc-accuracy.md「ドキュメント生成の正確性ルール」を厳守すること。
