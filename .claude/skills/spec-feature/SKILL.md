# 既存機能のSpec化

## フェーズ0: 統一基盤の確認

[Spec 統一基盤確認](../_shared/spec-unified-base.md) を実施する。

## フェーズ1: コード探索（構造把握）

1. 指定された機能グループに関連するコードを特定
2. [2段階探索](../_shared/code-search-2stage.md) で関連ファイルを全て把握
3. **関連機能の発見**: コード探索で見つかった依存関係から、関連する既存Specを特定する
   - 共有モデル・共有APIエンドポイントから関連Specを逆引きする
   - 特定した関連Specは要件定義書の「関連 Spec」セクションに記載し、`overview.md` の依存関係も更新する
4. 各ファイルの役割・行数・publicメソッド数を一覧化する

## フェーズ2: ビジネスロジック深掘り

構造把握だけでは仕様書にならない。このフェーズでコードの中身を正確に把握する。

### 全行読み
対象機能のサービスファイルをReadツールで先頭から末尾まで読む。ファイルが大きい場合は500行ずつoffset/limitで分割する。grepで断片的に読んで推測してはならない。

### コールチェーン追跡
エントリポイントから末端まで呼び出し先を追跡し、ファイル名:行番号を記録する。

```
processPayment()
  → validateOrder()       order.service.ts:120
  → calculateTotal()      pricing.service.ts:200
    → applyDiscount()     pricing.service.ts:250
```

### 共通コンポーネントの洗い出し
コード探索中に、複数機能で再利用されている共通コンポーネントを特定する:
- UIコンポーネント（ボタン、フォーム、モーダル等）
- ユーティリティ関数（バリデーション、フォーマッタ、APIクライアント等）
- カスタムフック（認証、データフェッチ等）
- 認証ガード、ミドルウェア等

各コンポーネントについて以下を記録する:
- コンポーネント名・種別・ファイルパス
- Props/パラメータのシグネチャ
- 利用箇所（どの機能で使われているか）

### 抽出する情報
各メソッドについて以下を抽出する:
- アルゴリズムの処理順序・分岐条件と分岐先
- 計算式・定数値・バッチサイズ・タイムアウト
- エラー条件・DB検索条件

## フェーズ3: 仕様書生成

フェーズ2の抽出結果をもとに生成する。全ての記述に根拠（`ファイル名:行番号`）を明記すること。

1. docs/requirements/features/[グループ名].md に要件定義書
   - REQ-ID採番、受入条件（具体的な数値を明記）、非機能要件
   - ビジネスロジック（アルゴリズム、条件分岐、計算式）をMermaid図で視覚化
   - **要件定義書にはWHAT（何を作るか）だけを記載する。** 画面遷移図・APIエンドポイント・データモデル等のHOW（どう作るか）は基本設計フェーズの設計書に記載済みのため、要件定義書には含めない。「関連設計書」セクションでリンクのみ記載する
2. [spec-map.yml 操作ガイド](../_shared/spec-map-operations.md) に従い、エントリを追加
3. mkdocs.yml の nav を更新
4. 必要に応じて基本設計（architecture.md, db-design.md, screen-flow.md, openapi.yaml）も更新。外部連携がある場合は api-spec.md（外部連携仕様書）も更新
5. **共通コンポーネント設計書の生成/更新:**
   - フェーズ2で洗い出した共通コンポーネントがある場合、`docs/design/shared-components.md` を生成または更新する
   - テンプレート（`docs/templates/phase2/shared-components.md`）のセクション構成に従う
   - 既に shared-components.md が存在する場合は、新たに発見したコンポーネントを追記する
   - feature-design.md のセクション3「共通コンポーネント」にも、利用する共通コンポーネントを記載する

## フェーズ4: 検証

- [ ] 各セクションに根拠（ファイル名:行番号）があるか
- [ ] 推測表現（「〜と思われる」「一般的に〜」）がないか
- [ ] コードに存在しない機能を書いていないか
- [ ] mkdocs build でエラー/警告がないか

## フェーズ4.5: 統一性レビュー

既存の Spec が1つ以上ある場合、[Spec 統一性レビュー](../_shared/spec-consistency-review.md) を実行する。

1. サブエージェントに全既存 Spec と今回作成した Spec の横断チェックを依頼する
2. レビュー結果への対応：
   - **自動修正可能** → 即座に修正
   - **要確認** → ユーザーに選択肢を提示
3. 用語統一テーブルがあれば `docs/requirements/glossary.md` を更新する

※ 初回の Spec（他に既存 Spec がない場合）はこのフェーズをスキップする

## フェーズ5: 完了

1. コミット（探索で見つけた全ファイル一覧をコミットメッセージに記録）
2. 次のステップを提案（「他の機能グループもSpec化しますか？」等）

## ルール

- **実装に書いていないことは書かない。** メソッド名や変数名だけで「こういう機能があるはず」と推測するのはNG。例: メソッド名だけ見て「包括的分析ダッシュボード」、行数だけ見て「Slack/SMS/Push通知システム」と書くなど
- **画面遷移図もコード根拠必須。** ルーティング定義やリンク先が確認できない画面・遷移は記載しない
- **Taskエージェントに仕様書作成を丸投げしない。** ファイル探索の補助には使ってよいが、仕様書の本文はメインエージェントが直接コードを読んで書くこと
- 既存コードは一切変更しない（spec-map.yml への記録のみ）

**コード分析時の注意:** .claude/rules/doc-accuracy.md「ドキュメント生成の正確性ルール」を厳守すること。
